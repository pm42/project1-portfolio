Flask - Micro Web Framework written in Python

Introduction

Flask is a web framework, it’s a Python module that lets us develop web applications easily. It has a small and easy-to-extend core: it’s a microframework that doesn’t include an ORM (Object Relational Manager) or such features. It does have many cool features like url routing, template engine. It is a WSGI web app framework.

A Simple Web Page Using Flask

  • First we imported the Flask class. An instance of this class will be our WSGI application
  • Next we created an instance of this class. The first argument is the name of the application’s module or package
  • We then used the route() decorator to tell Flask what URL should trigger our function
  • The function returns the message we want to display in the browser. The default content type is HTML, so HTML in the string will be rendered by the browser.

Static Files Rendering

Our web applications needs static files. That’s usually where the CSS and JavaScript files are coming from. Ideally our web server is configured to serve them for us, but during development Flask can do that as well.

For that we just need to create a folder called static in our package or next to our module, and it will be available at /static on the application.

To generate URLs for static files, we use the special 'static' endpoint name. The file has to be stored on the filesystem as static/style.css.

Rendering Templates

To render a template we use the render_template() method. All we have to do is provide the name of the template and the variables we want to pass to the template engine as keyword arguments

Flask will look for templates in the templates folder.

Blueprint Concept

The basic concept of blueprints is that they record operations to execute when registered on an application. Flask associates view functions with blueprints when dispatching requests and generating URLs from one endpoint to another.

When we bind a function with the help of the @simple_page.route decorator, the blueprint will record the intention of registering the function show on the application when it’s later registered.Additionally, it will prefix the endpoint of the function with the name of the blueprint which was given to the Blueprint constructor (in this case simple_page). The blueprint’s name does not modify the URL, only the endpoint.

Testing Flask Applications Using Pytest

Flask provides utilities for testing an application. We are using pytest framework to set up and run our tests

Tests are typically located in the tests folder. Tests are functions that start with test_, in Python modules that start with test_.

  • Client has methods that match the common HTTP request methods, such as client.get()
  • To make a request, we call the method the request should use with the path to the route to test
  • A TestResponse is returned to examine the response data. It has all the usual properties of a response object
  • We then check the response.data for a specific text to complete the validation
  • To run pytest, from Docker services open the docker flask application and then create terminal and run pytest

Project Layout

The project directory will contain:

  • Dockerfile

    File used to create an image to run in a container that runs the program

  • app - Package containing our application code and files
    • __init.py__ - Python file that creates and configures an instance of the Flask application
    • simple_pages
      • templates - Contains the .html files for our web page
      • __init__.py - Python file that registers the blueprint and contains code for routing and rendering the template
    • static
      • css - Folder containing the css files that are used in our web application
      • images - Folder containing the images that are used in our web application
      • js - Folder containing the javascript files that are used in our web application
  • docker-compose.yml

    Orchestration file that contains the configuration to develop locally, it overrides the Dockerfile to run the flask development server instead of running the gunicorn server that is used for hosting on Heroku<

  • heroku.yml

    A manifest file that allows us to build Docker images and specifies the release phase on Heroku

  • readme.md

    A markdown file that introduces and explains our project. It contains Github Workflow badges and Heroku URL

  • requirements.txt

    A text file specifying the python packages that are required to run our project

  • tests

    Contains python file that contains code to test our web page
