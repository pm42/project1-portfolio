SOLID Principles

Introduction

The SOLID principles were introduced by Robert C. Martin and were later built upon by Michael Feathers.

Martin and Feathers' design principles encourage us to create more maintainable, understandable, and flexible software. Consequently, as our applications grow in size, we can reduce their complexity and save ourselves a lot of time.

In this article, we will discuss the five concepts that makes up SOLID principles using my Calculator program.

Single Responsibility Principle

A class should have one and only one reason to change, meaning that a class should have only one job.

Single Responsibility Principle in Calculator

In calculator, each class handles its own responsibility.

Addition class handles the addition of two numbers and Multiplication class handles the multiplication of two numbers.

Open Closed Principle

Objects or entities should be open for extension but closed for modification.

Open Closed Principle in Calculator

Calculation class is closed for modification, but it is extended by Division class

Division class extends the Calculation class functionality to provide the results of the calculation.

Liskov Substitution Principle

Every subclass or derived class should be substitutable for their base or parent class.

Liskov Substitution Principle in Calculator

Division class here is inherited from Calculation class and it can readily substitute calculation class as it implements the abstract method get_result() of the parent class.

Interface Segregation Principle

A class should never be forced to implement an interface that it does not use, or class shouldnâ€™t be forced to depend on methods they do not use.

Interface Segregation Principle in Calculator

Calculation class contains only one abstract method get_result() which is of use to all the derived classes as it defines the functionality to retrieve the results of an operation

Apart from this abstract method, there are no other abstract methods in Calculation class which forces the derived classes to implement it.

Dependency Inversion Principle

Entities must depend on abstractions, not on concretions. It states that the high-level module must not depend on the low-level module, but they should depend on abstractions.

Dependency Inversion Principle in Calculator

Defining addition, subtraction, multiplication and division as individual methods under Calculation class would have made all the calculations tightly coupled with Calculator.

By creating each calculation as separate classes and making them dependent on abstraction method from Calculation class removed the dependency on low level module.
